全龄友好的无障碍写作准则

页面摘要：全球约 15% 的人口有无障碍需求。编写具有无障碍意识的文档，不仅能服务残障人士，更能提升所有读者的阅读体验。

info

本规范基于 Google Technical Writing Style Guide 中关于产品命名与引用的相关章节整理而成，并结合中文技术文档的实际写作场景进行了筛选与调整。 本文档旨在提供可执行的编辑规范，而非定义官方品牌使用规则。

1. 易读性与内容清晰度​

1.1 简明句式原则​

尽可能使用简短、直接的句子。对于中文技术文档，建议单句不超过30-40个汉字。过长的句子应拆分为多个短句。

✅示例（长句拆解）：

原句：在配置完成后，您需要重启服务以使更改生效，否则新设置将不会被加载。

优化：配置完成后，请重启服务。这样更改才能生效，新设置才会被加载。

1.2 拒绝“双重否定”​

说话要直接，不要绕弯子。

  • ✅推荐：你可以继续。
  • ❌避免：如果不点击这个按钮，你将无法继续。

1.3 禁用方位词​

不要说“点击右侧按钮”或“见下图”，因为对于盲人或使用屏幕阅读器的读者，这些方位信息是无效的。

  • ✅推荐：请参阅“前文”章节。
  • ❌避免：请参阅“下方”内容。

2. 文本内容规范​

2.1 链接文字要有意义​

链接需要让人一眼看出点进去是什么，不能依赖上下文。

  • ✅推荐：关于[链接命名规范]的详细说明。
  • ❌避免：点击[这里]了解更多。

2.2 避免缩写冲突​

有些阅读器会逐个字母读大写单词。除非该缩写家喻户晓（如 AI），否则应在首次出现时定义。

2.3 标点谨慎使用​

部分屏幕阅读器可能无法正确识别某些标点或特殊字符：

  • 避免：使用表情符号或特殊符号作为关键信息载体
  • 谨慎：依赖分号、冒号等区分列表项（可使用数字或项目符号）
  • 检查：数学符号、编程符号是否被阅读器正确朗读
  • ✅推荐：选项包括：1. 保存并退出 2. 仅保存 3. 取消
  • ❌避免：选项包括：保存并退出；仅保存；取消

2.4 语言声明与编码​

确保文档系统正确设置了：

  • 页面语言属性（如lang="zh-CN"）
  • UTF-8字符编码

3. 结构与导航​

3.1 语义化结构​

在使用Markdown时，确保正确使用标题层级（# ## ###），它们会自动生成对应的语义化HTML标签。

3.2 键盘导航友好​

确保读者无需鼠标，仅靠键盘就能操作所有按钮和标签。

3.3 跳过导航链接​

为提升键盘和屏幕阅读器用户的效率，每个页面坚持提供“跳过导航”功能，允许用户直接跳到主要内容。（通常由文档主题/模板实现，文档作者需确保内容结构支持此功能）

3.4 表格使用准则​

  • 谨慎使用：仅当信息本质上是表格数据时使用（如参数对比、功能矩阵）。
  • 结构清晰：建议有明确的表头<th>，Markdown中第一行作为标题。
  • 避免合并单元格：屏幕阅读器按行-列顺序读取表格。合并单元格会破坏这种线性路径，导致读者无法理解表格结构。
  • 提供摘要：复杂表格可考虑在表格前提供文字摘要，帮助读者理解内容。

4. 视觉与多媒体​

4.1 文本优先于图像​

请始终提供代码和配置参数的文本版本，避免使用截图。推荐使用真实文本，以便阅读器识别和搜索。

4.2 拒绝色彩依赖​

不能只靠颜色传递信息（如“红色代表错误”）。

  • ✅推荐：使用文字说明（如“错误：XXX”）或图标加文字。
  • ❌避免：错误信息显示为红色。

4.3 替代文本​

每张图片都建议提供有意义的替代文本。Alt text应当：

  • 装饰性图片：使用空alt文本（alt=""）或在代码中标记为装饰性。
  • 信息性图片：简要描述图片内容或功能（如“设置界面的截图”）。
  • ✅推荐：![网络拓扑结构图](topology.png)
  • ❌避免：![图片](topology.png)

4.4 多媒体的字幕与文稿​

建议提供字幕、文稿和音频描述。

5. 格式与排版​

5.1 左对齐​

永远保持文本左对齐，不要使用居中或两端对齐，因为那会产生不均匀的间距，干扰阅读。

5.2 SVG 优于 PNG​

尽可能使用矢量图（SVG），因为它们在放大后依然清晰，方便视力不佳的读者查看。

附录：无障碍文档快速检查清单​

  • 所有图片都有有意义的alt文本
  • 所有链接都有描述性文字
  • 标题层级不跳级
  • 不使用颜色作为唯一信息载体
  • 表格有明确标题且未合并单元格
  • 键盘可以访问所有交互元素
